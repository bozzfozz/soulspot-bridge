## Epic: v2.0 Dynamic Views & Widget-Palette

**Title:** `[EPIC] v2.0 — Dynamic Views & Widget-Palette`

**Labels:** `epic`, `v2.0`, `feature`, `ui`, `api`

**Description:**

Implementiere Dynamic Views & Widget-Palette als neue Hauptfunktionalität für SoulSpot v2.0. Ermöglicht Nutzern, personalisierte Dashboards mit konfigurierbaren Widgets zu erstellen.

**Kernfeatures:**
- Grid-basiertes Canvas mit Drag & Drop
- Widget-Palette mit 5 Core Widgets
- Save/Load Views (DB-persistiert)
- Composite Widgets (Parent mit children)
- Permission-System für Widget-Actions
- WebSocket für Real-Time Updates

**Timeframe:** Q3-Q4 2025  
**Effort:** ~20-26 dev days (4-5 sprints)

**Milestone:** v2.0.0

**Related Documentation:** [Development Roadmap - v2.0 Section](../docs/development-roadmap.md#-v20--dynamic-views--widget-palette-geplant)

## Phase B: Infrastructure MVP

### Issue B1: Grid Canvas + Widget-Palette Backend/Frontend

**Title:** `v2.0/infra: Grid Canvas + Widget-Palette (MVP infra)`

**Labels:** `v2.0`, `phase-b`, `infra`, `backend`, `frontend`, `api`

**Description:**

Implementiere Grid-Canvas UI mit Drag & Drop und Widget-Palette Backend/Frontend.

**Scope:**

**Backend:**
- Widget-Registry API: `GET /api/widgets`
- Views Management API:
  - `GET /api/views` – List user views
  - `GET /api/views/:id` – Get view
  - `POST /api/views` – Create/update view
  - `DELETE /api/views/:id` – Delete view
- Database schema für Views (Alembic Migration)
- Widget-Registry Service (load from config/registry)

**Frontend:**
- Grid-Canvas Component (12-column grid)
- Drag & Drop Integration (mit Collision Detection)
- Widget-Palette Component (zeigt verfügbare Widgets)
- View Management UI (Create, Save, Load, Delete)
- Responsive Layout (Tablet/Desktop)

**API Contracts:**

```http
GET /api/widgets
Response: {
  "widgets": [
    {
      "id": "widget-id",
      "name": "Widget Name",
      "description": "...",
      "category": "monitoring",
      "defaultSize": { "w": 4, "h": 3 },
      "settingsSchema": { ... },
      "actions": [ ... ]
    }
  ]
}

GET /api/views
Response: {
  "views": [
    { "id": "view_123", "name": "My Dashboard", "widgetCount": 5 }
  ]
}

POST /api/views
Request: {
  "name": "My Dashboard",
  "layout": { ... },
  "widgets": [ ... ]
}
Response: { "id": "view_123", "name": "My Dashboard" }
```

**Acceptance Criteria:**
- [ ] Grid-Canvas rendert korrekt (12-column Grid)
- [ ] Widgets können per Drag & Drop platziert werden
- [ ] Collision Detection funktioniert
- [ ] Widget-Palette zeigt alle verfügbaren Widgets
- [ ] View kann gespeichert werden (DB-Persistierung)
- [ ] View kann geladen werden
- [ ] View kann gelöscht werden
- [ ] Grid-Layout ist responsive (Tablet/Desktop)
- [ ] User sieht nur eigene Views

**Definition of Done:**
- [ ] Backend-Endpoints implementiert (GET/POST/DELETE /api/views, GET /api/widgets)
- [ ] Unit Tests für alle API-Endpoints (Coverage > 80%)
- [ ] Integration Tests für Views CRUD

### Issue D2: Permissions & Role Checks

**Title:** `v2.0/security: Permissions & role checks for widget actions`

**Labels:** `v2.0`, `phase-d`, `security`, `permissions`, `backend`

**Description:**

Implementiere vollständige Permission-Prüfung für alle Widget-Actions.

**Scope:**

**Backend:**
- Role Model implementieren (admin, curator, user, readOnly)
- Permission-System erweitern
- Serverseitige Validierung für alle Actions
- Audit-Log für Permission-Denied-Fälle

**Frontend:**
- Permission-basiertes UI (Buttons disablen wenn keine Permission)
- Error-Handling für Permission-Denied

**Role Model:**
- `admin` – Volle Kontrolle
- `curator` – Kann eigene Views verwalten, Read-Access auf shared Views
- `user` – Kann eigene Views verwalten
- `readOnly` – Nur Lesezugriff

**Permissions:**
- `views:read` – Views lesen
- `views:write` – Views erstellen/ändern
- `views:delete` – Views löschen
- `views:share` – Views teilen (Phase F)
- `jobs:read` – Jobs lesen
- `jobs:write` – Jobs steuern (pause, cancel, retry)
- `library:read` – Bibliothek lesen
- `library:write` – Bibliothek ändern
- `settings:read` – Einstellungen lesen
- `settings:write` – Einstellungen ändern (nur admin)

**Acceptance Criteria:**
- [ ] Role Model dokumentiert und implementiert
- [ ] Alle Widget-Actions prüfen Permissions
- [ ] Frontend zeigt nur erlaubte Actions
- [ ] Audit-Log für Permission-Denied-Fälle
- [ ] Permission-Denied gibt aussagekräftige Fehlermeldung

**Definition of Done:**
- [ ] Permission-System vollständig implementiert
- [ ] Unit Tests für Permission-Checks (alle Actions) (Coverage > 80%)
- [ ] Integration Tests für verschiedene Rollen
- [ ] E2E-Test: User mit readOnly, User mit admin
- [ ] Security-Review abgeschlossen
- [ ] Dokumentation aktualisiert (`docs/security/permissions.md`)
